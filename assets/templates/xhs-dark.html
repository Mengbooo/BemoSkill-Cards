01 · 架构感

先分层，再写代码

把页面拆成展示、数据、交互、边界四层。你会更早发现：哪里该复用，哪里该隔离。

1

展示层只关心可见状态

不要把请求、缓存和权限塞进每个组件。

2

业务逻辑有稳定入口

hook 可以复用，但边界要比技巧更重要。

3

异常路径也要被设计

loading、empty、error 是产品体验的一部分。

03 · 组件设计

组件不是越小越好

好组件应该降低理解成本。拆分的标准不是行数，而是职责、变化原因和复用边界。

应该拆变化原因不同，测试边界清晰
不必拆只是为了少看几行代码
应该暴露少量稳定、可组合的 props
需要隐藏内部状态、实现细节、临时变量

04 · 状态管理

状态越少，系统越稳

不要急着选库。先区分本地状态、服务端状态、URL 状态和派生状态。

L

Local

只影响一个局部交互，留在组件附近。

S

Server

来自接口的数据，需要缓存、失效和重试。

U

URL

筛选、分页、搜索词，应该可以分享和恢复。

05 · 接口契约

接口不是拿到数据就完事

前端要关心契约：字段是否稳定、错误是否可解释、分页是否一致、权限是否可预期。

type ApiResult = | { ok: true; data: Data } | { ok: false; message: string } UI 只消费被整理过的数据。

06 · 工程化

把重复劳动交给工具

格式化、类型检查、lint、测试、构建预检，都应该在提交前暴露问题。

A

自动化

能机器检查的，就不要靠人肉记忆。

C

可观察

线上问题要能定位，而不是只会刷新重试。

08 · AI 编程

会提需求，比会贴报错更重要

把目标、边界、验收标准、现有代码风格讲清楚，AI 才能成为工程搭档。

1

给上下文

说明业务目的、相关文件和不能改的边界。

2

给验收

告诉它什么叫完成，而不是只说“优化一下”。

3

要复核

AI 能提速，但你仍然要负责判断。

09 · 最后的检查清单

写完代码后，问自己 5 个问题

1

用户失败时看见什么？

空状态、错误态、加载态是否完整。

2

下个人能不能接手？

命名、边界、提交记录是否清楚。

3

上线后怎么知道坏了？

日志、监控、回滚路径是否准备好。
